# 分段字幕优化窗口以提升子序列通过率

字幕优化（见 [ADR 0011](0011-optimize-clip-subtitles-with-llm-under-subsequence-constraint.md)）真实端到端跑通后发现：5 条短视频里 4 条画面无烧录字幕。根因是 `step-3.7-flash` 在子序列约束下大量违规——用空格当显示块分隔符（增字）、改写造词、语序重排——导致 `validate_and_align` 校验失败、回退抑制烧录。本 ADR 修订 0011 的「作用范围」（Q2）与「提示词形态」（Q6），把「每条 clip 窗口跨 chunk 一次调用」改为「按字符预算分段、逐组调用」，并把提示词从「优化/整理」改为「抽取式删减（橡皮擦式：显式禁空格 + 禁同义替换/语序调整 + few-shot）」。

## 实验数据（已用真实素材验证）

三个杠杆叠加，子序列通过率 1/5 → 5/5：

| 阶段 | 通过率 |
|---|---|
| 原始（旧提示·high·temp0.2·整窗口） | 1/5 |
| 新提示（抽取式 + 禁空格 + few-shot） | 2/5 |
| + `reasoning_effort=low` + `temperature=0` | 3/5 |
| + 窗口字符预算 ≈ 100 分段逐组请求 | 5/5 |

## Considered Options

- **作用范围（修订 0011 Q2）**：选「按字符预算分段、逐组调用」而非「整窗口一次调用」。0011 选择跨 chunk 是为把 ASR 硬断的半句合回完整句；这一点保留——分段仍在 clip 窗口内、按 chunk 贪心分组，半句不会被再次拆散。新增的约束是「每组文本不超过 ≈100 字符」：文本越短，模型重组语序、造词的空间越小，子序列违规概率越低。代价是单条 clip 请求数从 1 增至 4~6，但优化器按子窗口文本签名缓存，同一窗口只真实请求一次、跨运行复用，属一次性成本。
- **提示词形态（修订 0011 Q6）**：选「抽取式删减（橡皮擦隐喻）」而非「优化/整理」。把任务重定义为「只能从原文擦掉字符，剩下的逐字照抄」，显式列出绝对规则（禁新增任何字符尤其禁空格、禁改写/同义替换/语序调整/补全），并附 few-shot 演示删语气词 + 换行分块。措辞从「整理成字幕」转向「只做删除」，消灭增字/空格类违规。
- **分组粒度**：贪心按字符预算分组，单 chunk 超额则独占一组——保证既不破坏 chunk 内连续性，又对超长 chunk 有确定行为。
- **默认预算**：`subtitle_optimization_window_max_chars = 100`。该默认下既有短窗口（窗口文本 < 100 字）仍是单组、单次请求，行为不变、零回归。
- **缓存粒度**：仍按子窗口文本签名（`_cache_signature` 含 text + model + base_url + 切分参数 + `PROMPT_VERSION`）。`window_max_chars` 不进签名——它只决定如何切，切出的 text 已在签名里。提示词改抽取式触发 `PROMPT_VERSION` v1 → v2，旧缓存自动失效。
- **本地调优参数**：`reasoning_effort=low` + `temperature=0` 写入 `config.local.json`（被 `.gitignore` 忽略），不改 `config.py` 全局默认，避免影响其它模型/场景。

## Consequences

- 失败回退策略不变：任一组 HTTP/超时/非子序列失败仍整体判 `FAILED`，该条短视频不烧录字幕、旁挂规则 SRT、标人工复核（保持 0011 的强依赖 + 确定性兜底契约）。
- 子窗口缓存使分段的请求放大成为一次性成本；缓存命名空间、签名构成与 0011/0009 一致，不污染语音识别与主题评审缓存。
- 提示词 bump 缓存版本后旧 v1 缓存失效，首跑会按新签名重新请求并落盘。
- `window_max_chars` 是可调旋钮：调小提升通过率但增加请求数与缓存条目，调大相反；默认 100 是该素材上通过率与成本的平衡点。
